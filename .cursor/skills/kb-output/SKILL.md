# kb-output スキル — 統合アウトプット

## いつ使うか

**KBを「使う」操作はすべてこのスキル。ユーザーはスキルを選ばなくてよい。**

| ユーザーの入力 | このスキルが対応 |
|---|---|
| 「[プロジェクト]についてまとめて」 | KB参照 → 構造化要約を生成 |
| 「壁打ちしたい」「考えを整理したい」 | knowledge-rally に委譲 |
| 「[部品]の選定について教えて」 | KB 横断検索 + 要約 |
| 「AとBの違いを比較して」 | KB参照 → 比較分析を生成 |
| 「このドキュメントを更新して」 | 対象ファイル読み込み → 編集提案 |
| 「HTML で出力して」 | html-export に委譲 |

## 参照スキル

- `knowledge-rally`: 壁打ち・思考整理ラリー
- `html-export`: HTML 単一ファイル出力

---

## Step 0: 目的の自動判定

| キーワード | 判定モード |
|-----------|------------|
| 「壁打ち」「考えを整理」「仮説を試したい」「ラリー」 | **rally** |
| 「HTML で出力」「シングルファイル」「html-export」 | **html** |
| 「まとめて」「分析して」「比較して」「教えて」 | **analyze** |
| 「更新して」「修正して」「追記して」「編集して」 | **edit** |

---

## Step 1: 自動素材収集

```
# research_map.md でテーマに関連するドキュメントを特定
# L0 を一括読みして関連度を判定
# 必要な L1/L2 を深掘り読み込み
```

収集順序:
1. `knowledge/00_index/research_map.md`
2. 関連プロジェクトの `knowledge/10_projects/<project>/README.md`
3. 該当する canonical ドキュメント（L0 → L1 → L2）
4. 必要なら `20_components/` `30_standards/` `40_suppliers/` `50_research/`
5. 個人視点が必要なら `80_sandbox/`

---

## Step 2: 目的確認

処理前にユーザーに確認:

```
モード: [analyze / rally / html / edit]
対象: <要約・分析するテーマ or 編集対象ファイル>
参照範囲: [関連ファイル列挙]

この内容で進めますか？
```

---

## Step 3: 生成・実行

### rally モード

`knowledge-rally/SKILL.md` の手順に完全委譲。

### analyze モード

KB 参照を元に構造化出力を生成:

```markdown
## 概要（L0相当）
**[核心的主張を1文]**

## 主要ポイント（L1相当）
- **[キー1]**: [説明]
...

## 詳細分析（L2相当）
### [セクション1]
[詳細内容]

## 参照したドキュメント
- [[doc_id]] — [one_line_thesis]
```

### html モード

`html-export/SKILL.md` に完全委譲。

### edit モード

対象ファイルを読み込み、変更提案 → 承認後に実行 → `as_of` を更新 → CHANGELOG 追記。

## 使わないケース

- **何かを保存したい** → `kb-intake`
- **議事録から決定・タスクを抽出したい** → `compile-meeting`
- **要件を grill したい** → `requirements-grill`
