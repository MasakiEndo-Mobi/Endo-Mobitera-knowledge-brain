# daily-log スキル — 日報構造化

## いつ使うか

| ユーザーの入力 | このスキルが対応 |
|---|---|
| 「日報を書きたい」+ 自然文 | 新規 daily-log を生成 |
| 「`<path>` の日報を取り込んで」 | 既存ファイルを構造化・標準フォーマットに整える |
| 「今日のタスクと進捗を整理」 | テキストから4セクション抽出 → daily-log 化 |
| 「明日の予定をタスクにして」 | 既存 daily-log から compile-meeting へエスカレーション |

## 使うべきでない場面

- 議事録（会議メモ）の取り込み → `kb-intake` または `compile-meeting`
- 個人着想・アイデアのメモ → `kb-intake`（inbox/）
- 設計検討メモ → `kb-intake`（design-notes/）

---

## 実行手順

### Step 0: プロジェクト確定

ユーザーがプロジェクトを明示していなければ確認:
```
どのプロジェクトの日報ですか？
利用可能: [10_projects/ の slug 一覧]
```

入力テキストから推測できる場合はその候補を先に提示。

### Step 1: 日付確定

ユーザーが日付を明示していなければ今日の日付を採用:
- 既定: `YYYY-MM-DD = 今日`
- ユーザーが「昨日」「2026-05-27」等を明示すればその日付

`10_projects/<project>/daily-logs/<date>.md` が既存ならば「上書き / 追記 / キャンセル」を確認。

### Step 2: 構造化抽出

ユーザーが提供したテキストを 4 セクションに振り分け:

```
今日のタスク       ← 完了/着手したタスクの列挙
進捗               ← 各タスクで何が進んだか
課題・気づき       ← 詰まった点・新しい発見・要検討事項
明日の予定         ← 次回やること
```

### Step 3: 確認ゲート

```
=== 日報プレビュー ===
ファイル: knowledge/10_projects/<project>/daily-logs/<date>.md

--- フロントマター ---
doc_id:           ops.daily.<project>.<date>
title:            日報 <date>
doc_type:         daily-log
project:          [<project>]
layer:            raw
status:           draft
as_of:            <date>
audience:         [self]
one_line_thesis:  <date> の作業ログ（<主要トピック>）

--- 本文プレビュー ---
## 今日のタスク
- ...

## 進捗
- ...

## 課題・気づき
- ...

## 明日の予定
- ...

このまま書き込みますか？ y/n/q
```

### Step 4: ファイル生成

確認後、上記内容で Write する。

### Step 5: タスク抽出の提案

書き込み完了後:

> この日報の「明日の予定」セクションには N 件のタスク候補が含まれています。
> `compile-meeting` で `10_projects/<project>/tasks/` に TASK-NNNN ファイルとして抽出しますか？

「はい」なら `compile-meeting` に処理を委譲（日報をソースとして入力）。

---

## 出力契約

このスキルは以下を必ず返す:

1. 生成ファイルのパス
2. 抽出した 4 セクションの行数サマリー
3. 「明日の予定」からの抽出候補タスク件数
4. compile-meeting への次のアクション提案（タスク抽出するか）

---

## Graceful Degradation

| 状況 | 対応 |
|---|---|
| 4 セクションのうち一部が空 | 該当セクションは「（なし）」と明示して書き込む |
| 「今日のタスク」と「進捗」の区別がつかない | 両セクションに同じ箇条書きを入れ、ユーザーに確認を求める |
| ユーザーが「3 日分まとめて」と複数日を提示 | 1 日ずつ分割して個別ファイルとして保存 |
| 既存ファイルとの差分が大きい | 上書きせず、差分プレビューを提示 |

---

## 禁止事項

- 議事録と混同して `meetings/` に保存しない
- 「今日のタスク」を勝手に推測して埋めない（ユーザー提供の事実のみ）
- `compile-meeting` を確認なしに自動実行しない
- 1 ファイル 1 日付の原則を破らない（同日の追記は同じファイルに集約）
