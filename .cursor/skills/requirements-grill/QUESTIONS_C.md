# QUESTIONS_C.md — Capability 層 定型問い

## この層の目的

「このシステムが何をできるか」を**実装非依存の能力単位**で列挙・定義する。
ここを厚くすることで、Feature や EngSpec が変わっても Capability は安定する。
「どの材料を使うか」「どの規格を使うか」は一切書かない。

## 完了条件（Exit Criteria）

各 Capability が独立した 1 ファイル（`CAP-NNN_<slug>.md`）として書き出せる状態。
最低 3 つの Capability が定義されていること。

## このあと進む層

Capabilities 完了 → Features（F）

## 書き出し先

`knowledge/10_projects/<project>/requirements/capabilities/CAP-NNN_<slug>.md`

## 定型問い一覧

### C-1: できること動詞列挙

**問い**: このシステムが「できるべきこと」を動詞で列挙してください。
（実装ではなくシステムの能力として。「〇〇できる」形式）

**車載機構設計の例**:
- 「振動下で〇〇を保持できる」
- 「温度サイクル下で気密を維持できる」
- 「組付け工程で N 秒以内に位置決めできる」
- 「整備時に工具なしで分解できる」

**推奨回答の探し方**:
1. 既存 design-notes から能力動詞を抽出
2. なければ PROMPT_RECIPES.md の機構設計能力動詞リストから候補提示

**escape hatch**: 10 以上 → 「最初のフェーズで必須なものを 3〜5 に絞ってください」

---

### C-2: 物理原理 vs 標準部品

**問い**: それぞれの capability は標準部品で実現できますか？
または新規設計／カスタム部品が必要ですか？

**推奨回答ガイド**:
- 「位置決め」「締結」→ 標準ネジ・スナップフィット等で可能なことが多い
- 「特定振動帯域減衰」「形状記憶」「電熱結合」→ カスタム設計が必要な場合が多い

**Web Search Trigger**: 標準部品ベンダーの最新カタログ
- クエリ例: `"<部品種別> automotive standard part 2026"`

**escape hatch**: 「わからない」→ [X] 保留。検証後に判断

---

### C-3: 依存前提（環境条件・周辺部品）

**問い**: 各 capability が機能するために必要な前提条件は何ですか？
（温度範囲・振動条件・周辺部品の前提・組付け手順等）

**推奨フォーマット**: 「〔capability名〕: 〔前提条件〕が必要」を列挙

**車載機構の例**:
- 「気密維持 capability: 周辺ハウジングの平面度 0.1mm 以下が前提」
- 「振動下保持 capability: ボルト締結トルク 8±1 N·m が前提」

**escape hatch**: 「特にない」→ [A] で「標準環境のみ」と記録

---

### C-4: 安定性（Stability）

**問い**: 各 capability の安定性（stable / evolving / volatile）は？

**推奨判定基準（車載機構）**:
- stable: 物理原理に近い（締結・位置決め・気密）
- evolving: 法規・規格改定で変わる（試験項目）
- volatile: サプライヤー部品依存（コネクタ仕様等）

**escape hatch**: 「すべて evolving」→ 粒度を変えるか、安定部分を分離

---

### C-5: Graceful Degradation

**問い**: 各 capability が失敗したとき、システムはどう振る舞うべきですか？

**推奨回答例**:
- 「シール 1 段失敗時 → 2 段目で防水維持、保証期間内に検出」
- 「ブラケット 1 本緩み → 残りで機能維持、警告表示」
- 「コネクタ片側脱落 → 冗長系で動作継続」

**escape hatch**: 「失敗を想定していない」→ 「フェイルセーフ設計の観点で何を担保しますか？」

---

### C-6: 独立進化できるか

**問い**: どの capability が他と独立して進化できますか？

**推奨**: capability 間の依存グラフを描き、「A が変わると B も変わる」関係を明記

**escape hatch**: 「全部依存」→ 「最も小さな独立単位は？」

---

### C-7: 複数プロジェクト間での共通化

**問い**: この capability が他プロジェクトでも使えそうなものはありますか？

**推奨**: 共通化できるものは `20_components/<category>/` への昇格候補

**escape hatch**: 「わからない」→ [X] 保留

---

### C-8: Outcome との紐付け

**問い**: 各 capability は、先ほど定義した Outcomes のどれにどう貢献しますか？

**推奨フォーマット**:
```
〔capability名〕 → 〔Outcome指標名〕に〔どう貢献〕
例: 「振動下保持 → 市場クレーム率（締結緩み起因）の削減」
```

**escape hatch**: 紐付かない capability → 「これはどの成功指標に影響しますか？影響しないなら今回のスコープに含めますか？」

---

## Escape Hatch 共通ルール（C 層）

- Feature の話 → 「それは F 層。Capability は実装非依存に」
- Engineering の話 → 「実装は S 層。Capability では『何ができるか』だけ」
- 粒度が大きすぎる → 「もう少し小さな動詞に分割を」
