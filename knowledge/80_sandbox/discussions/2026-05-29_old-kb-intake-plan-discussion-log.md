# 議論ログ：旧KB(from_old_kb)取り込み計画

## 素材の現状（議論の前提）
- `from_old_kb`：150 Markdown ＋ 画像24点 ＋ PDF1点
- 構造：PARA（`01_Projects` / `02_Areas` / `03_Resources`）
- 旧フロントマター：`type / status / priority / tags / created`（軽量）。現行KBの `doc_id`＋11必須＋L0/L1/L2 とは非互換
- `01_Projects`：21CY/24CY系の十数案件。現役は実質 825D のみ
- `02_Areas`：2025/04〜2026/03 の月次業務レポート、キャリア・学習・副業・保険等の個人メモ
- `03_Resources`：TOYOTA用語集、開発プロセス、DRBFM、3Dプリンター、サプライヤ、プロンプト等（陳腐化耐性高）

## 登場人物
- 🧑‍💼 北原（ナレッジ移行・情報アーキテクト）：KB移行/IA歴12年。レガシー資産の選別移行が専門。
- 🧑‍🔧 桑田（車載機構設計の実務者）：車載IVI/HMI機構設計歴18年。トヨタ系開発プロセスに精通。
- 💅 みさき（ギャル）：IT素人だが感性が鋭い。実際のObsidian KBユーザー代表。

## Turn 1｜現状認識：これは「移行」なのか？
- 北原：スキーマ非互換が全150ファイルに一律存在（旧軽量 vs 現行重量）。
- 桑田：賞味期限の非対称が重要。現役は実質825Dのみ、用語集/技術知見は陳腐化しない、月次は時系列保全価値。
- みさき：「生鮮食品と缶詰が同じ箱に混ざってる」状態。
- 🩺 診断：これは一括移行ではなく **「賞味期限 × 正本価値」で層別する選別を伴う再構造化** 問題。真の敵は変換の手間でなく「何を canonical に上げ/raw に留め/捨てるか」の判断コスト。canonical まで仕上げる価値があるのは肌感2〜3割。

## Turn 2｜選択肢の発散：取り込み戦略を軸の上に並べる
評価軸＝①投入する手間 ②現行KBへの統合度。

| 方式 | 手間 | 統合度 | 向く対象 |
|---|---|---|---|
| A. 一括ダンプ | 低 | 低（90_sources/raw/） | 終了案件・履歴保全 |
| B. 選別raw取り込み | 中 | 中（最小フロントマター＋raw層） | 用語集・技術調査 |
| C. canonical昇格 | 高 | 高（L0/L1/L2＋relations＋doc_id） | 825D・再利用正本 |
| D. オンデマンド遅延移行 | 都度 | 都度 | 使う時に kb-intake で昇格 |

- 排他でなく対象ごとに使い分け。
- みさき：「使う時に昇格」が現実的、150を今全部磨くのは心折れる。
- 北原：「全部を今やる」は移行アンチパターン。だが D 単独は発見性が落ちる → **B（索引）＋ D（昇格は遅延）のハイブリッドがスイートスポット**。

## Turn 3｜要件と範囲の確定：何を最優先で取り込むか
- みさき実需：TOYOTA用語集（外設申/支給/RDDP等をAIが理解しないと議事録要約がズレる）、月次レポート（本人の振り返り）。
- 桑田：現役で効く群＝用語・プロセス基盤／825D現役知見／再利用技術知見（DRBFM・3Dプリンター・cfrtp・サプライヤ）。
- 北原：優先度を支配する原則＝**「参照頻度 × 陳腐化耐性」が高いものから**。

| 優先度 | 対象 | 方式 |
|---|---|---|
| 🔴第1波 | TOYOTA用語集・開発プロセスガイド | B（00_index/ or 30_standards/internal/） |
| 🔴第1波 | 825D関連 | B→C（10_projects/24cy-ivi-825d/、現役は昇格） |
| 🟡第2波 | 技術調査・サプライヤ・DRBFM | B（50_research/ 40_suppliers/ 20_components/） |
| 🟡第2波 | 月次レポート | B（daily-logs相当/Areas相当） |
| 🟢第3波 | 終了案件（21CY系・他24CY） | A（90_sources/raw/ 履歴保全） |
| ⚪判断要 | キャリア/学習/副業/保険等の個人メモ | KBスコープ境界の論点（要ユーザー判断） |

## Turn 4｜技術深掘り：スキーマ変換の実務と支配法則
変換コストの非対称性（3層）：
1. 機械層：`type→doc_type`、`created→as_of`、tags引き継ぎ。一括自動化可、コスト低・線形。
2. 判断層：doc_id採番・project割当・layer判定・role_in_story。1件ずつ判断、選別と同時にやれば二度手間消滅。
3. 創作禁止層：one_line_thesis・confidence・L0/L1/L2。中身を読まないと書けない。コスト高 → 昇格対象（2〜3割）に限定。

- みさき：`kb-intake` にフォルダごと食わせれば？ → 北原：核心。第1・2層は kb-intake のディレクトリ取り込みに乗せ、第3層は対象を絞って後追い。
- 注意点：
  - raw層（90_sources/raw/・80_sandbox/）は L0/L1/L2 免除 → 「検索可能化」と「正本化」を時間分離できる緩衝材。
  - 画像24点：旧は画像埋め込み記法（感嘆符＋二重角括弧）で本文埋め込み。現行は assets/ 強制＋日本語/スペース禁止名 → リネーム＆リンク張り替えが必要、一括変換で壊れやすい。
  - 月次のLoop長大URLは本文URLなので validator 対象外。
  - 旧来のタイトル参照型 wikilink と現行 doc_id 体系の差で一括変換時に最も壊れる → 第1波は張り替えず素材を置き、リンク整合は `obsidian-link` で後工程。

⚖️ 法則：変換は3層に分解。機械層=一括／判断層=選別と同時／要約層=昇格対象のみ。raw層を緩衝材に検索可能化と正本化を時間分離。

## Turn 5｜方針統合：旧KB取り込みの3原則
① **Triage Before Transfer**（移す前に選別）：「参照頻度×陳腐化耐性」で第1〜3波に層別、波ごとに方式を変える。
② **Index Now, Polish Later**（索引は今、研磨は後）：raw層を緩衝材に検索可能性を先に全体へ、L0/L1/L2 は遅延昇格。
③ **Automate the Mechanical, Judge the Semantic**（機械は自動・意味は人）：フロントマター機械変換と画像移送は一括、doc_id/layer/要約は判断に回し要約は昇格対象限定。

明日からの具体アクション：
1. スコープ確定：個人メモ（キャリア/副業/保険/学習）をKBに入れるか別管理かを先に決める。
2. 第1波素材を物理隔離：用語集＋825D関連＋主要技術/サプライヤ（≒30〜40ファイル）を抜き出す。
3. 第1波を `kb-intake` にディレクトリ投入：種別判定→確認ゲート→raw取り込み、doc_id/フロントマター付与。
4. 825D現役分だけ canonical 昇格：L0/L1/L2＋relations、`10_projects/24cy-ivi-825d/` 配下へ。
5. 第2・3波は後日/オンデマンド：終了案件は 90_sources/raw/ にダンプ、技術知見は必要時昇格。
6. 最後にリンク整合：`obsidian-link` で掃除、3バリデータを通す。
