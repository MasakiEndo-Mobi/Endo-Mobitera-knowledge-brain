# QUESTIONS_S.md — Engineering Spec 層 定型問い

## この層の目的

「どう作るか」を技術仕様として定義する。最も頻繁に変わる層。
Feature / Eval が固まった後に詰める。
製造・調達・検査が指示として使える精度で書く。

## 完了条件（Exit Criteria）

材料 / 寸法 / 公差 / 規格対応 / サプライヤーの骨子が書かれていること。
製造・品質部門への申し送り事項が明示されていること。

## このあと進む層

Engineering Spec 完了 → 全層 writeout 完了 → セッション completed

## 書き出し先

`knowledge/10_projects/<project>/requirements/engineering/` 配下の各ファイル

---

## 定型問い一覧

### S-1: 材料

**問い**: 主要材料は何を使いますか？（樹脂 / 金属 / シール材 / 接着剤など）

**推奨回答の探し方**:
1. 既存 design-notes から材料選定根拠を確認
2. なければ PROMPT_RECIPES.md の機構材料プロファイル を提示

**Web Search Trigger**: 材料の最新スペック・新材料
- クエリ例: `"automotive housing PBT GF30 spec 2026"` / `"FKM seal material datasheet 2026"`

**escape hatch**: 「まだ決めていない」→ [X] 保留、PROMPT_RECIPES のデフォルトを提示

---

### S-2: 主要寸法と公差

**問い**: 主要寸法と公差を教えてください。

**推奨フォーマット**:
```
寸法名: 公差付き値（例: 50 ±0.1 mm）
データム面: A / B / C
幾何公差: 平面度 0.05 / 平行度 0.1 等
```

**escape hatch**: 「わからない」→ Feature の F-6 受け入れ基準から逆算

---

### S-3: インターフェース寸法

**問い**: 周辺部品（コネクタ・ECU・取付面）とのインターフェース寸法と公差は？

**推奨フォーマット**:
```
インターフェース: <周辺部品名>
  寸法: <値±公差>
  位置: データム面 A から <距離>±<公差>
```

**escape hatch**: 「決まっていない」→ 「先方部品の図面 No.は？」

---

### S-4: 試験条件詳細

**問い**: 適用する規格試験の具体的な条件は？（E-7 をさらに詳細化）

**推奨フォーマット**:
```
試験項目: ISO 16750-3 §4.X
試験条件:
  温度: <範囲>
  振動: <加速度・周波数帯>
  時間: <X 時間 × N サイクル>
判定: <閾値・観察項目>
```

**Web Search Trigger**: 規格の試験条件詳細
- クエリ例: `"ISO 16750-3 vibration profile 2026"`

**escape hatch**: 「規格を確認」→ 30_standards/ から該当 standard-note を Read

---

### S-5: サプライヤー候補

**問い**: 部品サプライヤーの候補は？

**推奨選択肢**:
- 既存取引サプライヤー（社内データベース）
- 新規候補（評価サンプル要請段階）
- カスタム品（社内製造）

**Web Search Trigger**: 部品サプライヤー
- クエリ例: `"<部品種別> automotive supplier list 2026"` / `"<vendor name> product catalog 2026"`

**escape hatch**: 「未定」→ [X] 保留、40_suppliers/ の候補を参照

---

### S-6: 製造方法

**問い**: 製造方法と量産規模は？

**推奨選択肢**:
- 射出成形（樹脂部品）
- ダイキャスト（金属部品）
- プレス加工（板金）
- 切削加工（金属部品・小ロット）
- 3D プリント（試作・特殊形状）

**escape hatch**: 「まだわからない」→ 形状・量産規模・コスト要件から推奨を提示

---

### S-7: 検査方法

**問い**: 量産時の検査方法は？

**推奨フォーマット**:
```
検査項目: <項目名>
方法: 寸法測定 / 外観 / 機能試験
頻度: 全数 / 抜取り N/M
判定: <合格基準>
記録: <データ保存方法>
```

**escape hatch**: 「決まっていない」→ E-8 の試験データから派生

---

### S-8: ADR 候補（重要な技術選択の記録）

**問い**: このシステムで「採用 / 不採用」を記録すべき重要な技術選択はありますか？

**推奨トリガー**:
- 「A と B で迷った」過去経緯
- 将来の設計者が「なぜこれを選んだ？」と疑問を持ちそうな場合

**車載機構の典型**:
- 「FKM vs HNBR シール材選定」
- 「ボルト締結 vs 接着剤締結」
- 「樹脂ハウジング vs 金属ハウジング」

**escape hatch**: 「特にない」→ [A] で「ADR なし」と記録

## Escape Hatch 共通ルール（S 層）

- Capability の話に戻る → 「Capability は確定済み。実装の具体化を進めよう」
- 情報が多すぎる → ファイル分割（materials.md / interfaces.md / test_procedures.md 等）
- 「あとで決める」→ ADR 候補としてリストアップ
